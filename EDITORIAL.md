# Editorial Guide

This document is the editorial constitution of Materials Atlas.

It defines how future pages should be written, evaluated, maintained, and deleted.

## Mission

Materials Atlas is a curated atlas of Computational Materials Science.

It is not a wiki, bibliography, bookmark collection, personal knowledge system, or database. Its purpose is to help readers understand the field by showing relationships between concepts, methods, software, datasets, people, institutions, and research problems.

The atlas should reduce uncertainty. It should help a reader understand where they are, what matters, what came before, and what to learn next.

## Editorial Principles

Content earns structure.

Prefer explanation over enumeration.

Prefer one canonical resource over five alternatives.

Prefer primary sources over summaries.

Every sentence should reduce uncertainty.

Delete before adding.

Optimize for conceptual navigation, not accumulation.

The repository should still make sense in 2040.

## Every Page Should Answer

Every page should make its role clear by answering:

- Why does this exist?
- Why does it matter?
- Why now?
- What should the reader already understand?
- What becomes possible after learning this?
- Where should the reader continue next?

If a page cannot answer these questions, it is incomplete.

## Every Resource Should Justify

Every resource should explain:

- Why this resource?
- Why not another?
- When should it be used?
- Is it foundational or reference?
- What conceptual gap would remain without it?

A resource belongs only when its absence would make the atlas less clear.

## Relationship Vocabulary

Relationships should remain human-readable. Use this small vocabulary:

- Read after: prerequisite concepts or resources.
- Leads to: the next useful concept, method, or practice.
- Related: nearby ideas that clarify context.
- Implemented by: software or infrastructure that embodies a method.
- Maintained by: people, groups, or institutions that shape an ecosystem.

Do not introduce new relationship types unless the existing vocabulary cannot express a recurring editorial need.

## Editorial Style

Write like an editor producing a graduate-level reference work.

Avoid marketing.

Avoid hype.

Avoid AI-generated filler.

Avoid encyclopedic repetition.

Avoid unsourced reputation claims.

Use concise explanations that help readers make decisions.

## Deletion Rule

Every page should have a reason to exist.

If removing a page would not reduce understanding, the page should not exist.

If a resource no longer clarifies a domain, workflow, historical transition, or research question, remove it.

## Scope Control

Reject additions that exist only because they are popular.

Reject additions that duplicate existing understanding.

Reject speculative organization.

Reject placeholder pages.

Reject long lists when one canonical resource gives a clearer route.

New structure requires exceptional justification. Most future work should add or improve content inside the existing architecture.

For domain-specific lifecycle, readiness, and cross-reference rules, see [domains/README.md](domains/README.md).

## Maintenance Standard

When editing a page, check whether the change improves:

- clarity
- curation
- conceptual navigation
- editorial consistency
- long-term maintainability

If it does not improve at least one of these, do not make the change.
